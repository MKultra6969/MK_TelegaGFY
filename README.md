# TelegaGFY

`TelegaGFY` - Telegram-бот для модерации пользователей, связанных с Telega. Проект построен на `aiogram` и `Telethon`.

Бот использует два сигнала из референсной реализации в [Telega Detector.plugin](https://t.me/kvucoPlugins):

- наличие активной Telega-сессии прямо сейчас, если Telegram отдаёт флаг `unofficial_security_risk` (сейчас мы рассматриваем его как индикатор активной сессии именно в Telega, но в будущем Telegram может начать выдавать этот флаг и другим неофициальным клиентам/AppID, поэтому этот сигнал не гарантирует привязку исключительно к Telega);
- факт использования Telega в прошлом, который определяется через тот же OK Calls lookup flow. (На данный момент, это единственный абсолютно точный идентификатор Telega юзера)

## Что умеет бот

- отслеживает новых участников в группах и супергруппах через `aiogram`;
- отслеживает вступления через `Telethon` chat actions и кэширует `user_entity`, чтобы повторно использовать access hash при MTProto-проверках;
- постранично опрашивает admin log каналов через `Telethon`, корректно обрабатывает invite/join события и пережидает `FloodWait`, потому что Bot API не всегда надёжно показывает новые вступления в broadcast-каналах;
- хранит настройки модерации по каждому чату в SQLite;
- даёт владельцу чата панель настроек в личном чате с ботом;
- пишет расширенный console log по пользовательским взаимодействиям с ботом: команды, callback-и, membership updates, join-события и постановку в moderation queue.

Для каждого чата можно включать отдельные правила:

- банить, если у пользователя есть активная Telega-сессия;
- банить, если пользователь использовал Telega раньше;
- не банить автоматически, а просто отправлять владельцу чата личное уведомление со статусом:
  - `Имеет активную сессию`;
  - `Использовал Telega раньше`.

## Требования

- Python `3.12+`
- Telegram-бот с правом блокировки пользователей
- аккаунт Telegram для `Telethon`

## Настройка окружения

Скопируйте `.env.example` в `.env` и заполните значения.

Обязательные переменные:

- `BOT_TOKEN` — токен бота из BotFather
- `API_ID` и `API_HASH` — Telegram API credentials для `Telethon`
- `TELETHON_PHONE` — номер телефона аккаунта, через который работает `Telethon`

Основные опциональные переменные:

- `TELETHON_SESSION` — путь к файловой сессии `Telethon`
- `TELETHON_SESSION_STRING` — `StringSession`, если не хотите хранить файловую сессию
- `TELETHON_2FA_PASSWORD` — пароль 2FA для аккаунта `Telethon`
- `OWNER_USER_ID` — Telegram user id владельца бота; нужен для owner-команд `/check_user` и `/broadcast`
- `DATABASE_PATH` — путь к SQLite-базе
- `LOG_LEVEL` — уровень логирования
- `CHANNEL_ADMIN_LOG_POLL_SECONDS` — интервал опроса admin log каналов
- `LOOKUP_CACHE_TTL_SECONDS` — TTL кэша проверки на прошлое использование
- `LOOKUP_FAILURE_COOLDOWN_SECONDS` — пауза после неудачной lookup-проверки
- `DUPLICATE_JOIN_TTL_SECONDS` — окно дедупликации повторных join-событий

При первом запуске `Telethon` попросит код подтверждения в терминале. Если на аккаунте включена двухфакторная аутентификация, также потребуется пароль.

## Запуск через Docker Compose

1. Заполните `.env`.
2. Один раз авторизуйте `Telethon`:

```bash
docker compose run --rm auth
```

3. Запустите приложение:

```bash
docker compose up -d app
```

4. Посмотрите логи:

```bash
docker compose logs -f app
```

Compose-монтирования:

- `./data` — для SQLite
- `./sessions` — для файловой сессии `Telethon`

Если используете `TELETHON_SESSION_STRING`, шаг с `auth` можно пропустить.

## Как пользоваться

1. Добавьте бота в группу, супергруппу или канал.
2. Выдайте боту право на блокировку пользователей.
3. Убедитесь, что `Telethon`-аккаунт имеет достаточную видимость пользователей там, где нужна MTProto-проверка.
4. Откройте личный чат с ботом и отправьте `/start`.
5. Откройте `/settings` в личном чате с ботом.
6. Выберите нужный чат и включите подходящие правила:
   - бан за активную сессию;
   - бан за прошлое использование;
   - уведомление владельцу без автобана.

Панель настроек открывается только в личном чате. Команда `/start` в группах, супергруппах и каналах игнорируется, чтобы бот не публиковал служебный onboarding в общих чатах.

## Команды

- `/start` — открыть стартовое сообщение и быстрые ссылки на добавление бота;
- `/settings` — открыть панель настроек доступных чатов в личном чате;
- `/logs` — посмотреть последние события модерации по доступным чатам;
- `/info` — показать краткую информацию о боте, авторе, канале и GitHub-репозитории;
- `/check_user <user_id> [chat_id]` — ручная проверка пользователя владельцем бота, если задан `OWNER_USER_ID`;
- `/broadcast` — owner-only рассылка по пользователям, которые уже писали боту в личный чат.

## Ручная проверка пользователя

Если задан `OWNER_USER_ID`, владелец бота может вручную проверять пользователей в личном чате с ботом:

```text
/check_user <user_id> [chat_id]
```

- `user_id` — обязателен;
- `chat_id` — опционален, но помогает `Telethon` лучше разрешить сущность пользователя для проверки активной сессии.

## Рассылка пользователям

Если задан `OWNER_USER_ID`, владелец бота может запустить `/broadcast` в личном чате с ботом.

Поток работы:

1. Отправьте `/broadcast`.
2. Следующим сообщением отправьте текст, одиночное медиа или альбом.
3. Бот покажет предпросмотр в том же виде, в котором сообщение уйдёт пользователям.
4. Отдельным сообщением бот попросит подтверждение кнопками `Да` / `Нет`.

Получатели рассылки - только те пользователи, которые уже взаимодействовали с ботом в личном чате. Сам владелец бота из списка получателей исключается.

## Как работает уведомление без бана

Если в настройках чата включён режим уведомления без автобана, то при обнаружении одного или обоих статусов бот:

- не банит пользователя автоматически;
- находит `Owner` этого чата;
- отправляет владельцу личное сообщение с найденными статусами.

Этот режим нужен для администраторов, которые хотят сначала вручную проверить пользователя, а не блокировать его сразу.

## Логирование

По умолчанию бот пишет расширенный console log, в котором видно:

- входящие команды и обычные сообщения;
- callback-и из inline-кнопок;
- `chat_member` и `my_chat_member` события;
- join/invite события из `aiogram` и `Telethon`;
- постановку пользователя в moderation queue;
- ключевые решения модерации, ошибки lookup и ограничения Telegram API.

Уровень логирования задаётся через `LOG_LEVEL`.

## Ограничения

- Broadcast-каналы обрабатываются через `Telethon` admin log polling, потому что Bot API не гарантирует полную видимость вступлений.
- Для проверки `active-session` аккаунт `Telethon` должен видеть пользователя через MTProto. Если Telegram не отдаёт нужную сущность, статус может остаться неизвестным.
- Проверки membership/access уважают rate limits Telegram и используют backoff, но при очень большом числе чатов ответы панели `/settings` и `/logs` всё равно будут зависеть от скорости ответов Bot API.
- При первом запуске в Docker авторизация `Telethon` всё равно требует интерактивного `docker compose run --rm auth`, если не используется `TELETHON_SESSION_STRING`.
